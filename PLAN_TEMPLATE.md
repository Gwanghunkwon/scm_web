# PLAN

## 프로젝트 개요
- 프로젝트명: 경량 SCM 웹 시스템
- 한 줄 목표: 수요예측 → 생산 → MRP → 발주 → 재고를 한 사이트에서 가볍게 운용 가능한 SCM 구축
- 현재 단계: 초기 설계 및 프론트엔드 뼈대 구축
- 최종 마감일: [마감일 또는 미정]

## 요구사항 재정의 (확정)
- 수요예측, 생산계획, BOM, 재고, 발주를 **단일 웹 UI**에서 조회·입력·관리할 수 있어야 한다.
- 수요예측 → 생산계획 → MRP → 발주 → 입고 → 재고 반영까지의 **업무 플로우를 끊기지 않고** 수행할 수 있어야 한다.
- 품목/BOM/리드타임/재고/발주 데이터는 기본적인 검색·필터·정렬이 가능해야 한다.
- 초기에는 단일 공장/창고 기준으로 설계하되, 향후 다공장/다창고로 확장 가능해야 한다.
- 복잡한 APS/원가/회계는 제외하고, **경량·실무 친화적인 수준**에 집중한다.

## 주요 리스크
- 보안: 로그인/권한 관리가 미비하면 재고/발주 정보 유출 가능성이 있으므로, 최소한 역할 기반 권한(RBAC)을 적용해야 한다.
- 데이터 손실: 초기 도입 시 기존 엑셀/시스템에서 마이그레이션 과정에서 데이터 손실·불일치가 발생할 수 있다.
- 성능: MRP 계산이 품목 수 증가에 따라 느려질 수 있으므로, 배치/캐싱 전략을 고려해야 한다.
- 일정: 실제 사용자 요구사항 변경이 잦을 수 있어 초기 릴리즈 범위를 엄격히 관리해야 한다.

## 단계별 실행 계획
### 1단계: 요구사항·모델링·아키텍처
- 입력: 현재 업무 플로우, 기존 엑셀/시스템 양식, 이해관계자 의견
- 출력: 확정된 요구사항 목록, ERD, 아키텍처 다이어그램, 메뉴/화면 목록
- 완료 기준:
  - PLAN/CONTEXT/기획서/계획서 문서 정리
  - 핵심 테이블(품목/재고/BOM/수요예측/생산계획/발주) 설계 확정
  - 프론트엔드/백엔드 기술 스택 및 배포 방식 결정
- 리스크: 요구사항 추가·변경에 따른 범위 증가

### 2단계: 기초데이터·수요/생산·재고 구현
- 입력: 1단계 산출물, UI 와이어프레임, ERD
- 출력: 품목/BOM/리드타임/수요예측/생산계획/재고 UI 및 API, 기본 리포트
- 완료 기준:
  - 기초 데이터(품목/BOM/리드타임/창고) CRUD 완료
  - 수요예측·생산계획 화면 및 API 동작
  - 재고/입출고 기본 구조 및 조회 화면 동작
- 리스크: 기초 데이터 구조가 이후 MRP·발주 요구와 충돌할 수 있음

### 3단계: MRP·발주·대시보드·운영
- 입력: 2단계까지 구현된 기능, 실제 샘플 데이터
- 출력: MRP 계산, 발주 계획/발주서 관리, 대시보드 및 리포트, 운영용 설정
- 완료 기준:
  - MRP 결과 기반 발주 계획/발주서 생성이 가능
  - 대시보드에서 재고/발주/계획 지표를 한 눈에 볼 수 있음
  - 기본 모니터링/백업/롤백 전략 수립 및 테스트
- 리스크: 성능(대용량 계산), 사용자 교육·도입 저항

## 완료 기준 (Definition of Done)
- [ ] 수요예측 → 생산계획 → MRP → 발주 → 입고 → 재고 반영 시나리오가 하나의 시스템 내에서 끊김 없이 수행된다.
- [ ] 핵심 역할(생산/구매/재고 담당자)이 일상 업무를 이 시스템에서 처리할 수 있다고 판단한다.
- [ ] 주요 화면(대시보드, 품목, BOM, 수요예측, 생산계획, MRP, 발주, 재고)에 대한 기능·성능 테스트가 통과한다.
- [ ] 운영·장애 대응·백업/복구 절차 문서가 준비되어 있다.
