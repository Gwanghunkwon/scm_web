# 경량 SCM 시스템 기획서

## 1. 프로젝트 개요
### 1.1 프로젝트명
경량 SCM(Supply Chain Management) 웹 시스템

### 1.2 프로젝트 목적
- 재고, 생산, 자재, 발주, 수요예측을 **하나의 웹 사이트에서 통합 관리**하여 업무 효율을 높인다.
- 수요예측 → 생산계획 → BOM 계산 → 자재 소요 계획(MRP) → 자재 발주 → 재고 관리 → 납품까지의 **엔드투엔드 흐름을 가시화**한다.
- 엑셀/수기 작업을 줄이고, **데이터 기반 의사결정**(재고/발주/생산)을 지원한다.

### 1.3 프로젝트 범위
**In Scope (포함 범위)**
- 품목, 재고, BOM(배합비), 리드타임, 생산계획, 발주, 수요예측 기초 데이터 관리
- 수요예측 등록 및 참고용 과거 실적 조회
- 생산계획 수립(예측 기반/수동 입력)
- BOM 전개를 통한 자재 소요량 계산
- 재고/리드타임/안전재고를 반영한 MRP 및 자재 발주 계획
- 발주서 생성/상태 관리 및 입고 연동
- 재고 현황 및 입출고 이력 관리
- 기본 대시보드 및 리포트(재고, 발주, 계획 vs 실제)

**Out of Scope (제외 범위)**
- 상세 APS(세부 일정 계획, 작업장·설비 단위 스케줄링)
- 회계/원가 계산, ERP 통합
- 물류/운송 최적화, TMS(WMS 연동은 향후 과제로 고려)
- 모바일 전용 앱(초기에는 웹 브라우저 기반으로 사용)

## 2. 기능 요구사항
### 2.1 기능 목록
- **인증/사용자 관리**
  - 이메일/비밀번호 기반 회원가입·로그인
  - JWT 기반 세션 유지 및 로그아웃
  - 역할(관리자/생산/구매/재고/조회 전용)에 따른 메뉴 접근 제어
- **기초 데이터 관리**
  - 품목 마스터(제품/원자재, 단위, 안전재고, 기본창고, 리드타임 등) 관리
  - BOM 관리(제품별 원자재 구성 및 소요수량)
  - 창고/로케이션, 거래처(공급사) 관리
- **수요예측 관리**
  - 기간(일/주/월)·제품별 수요예측 수량 등록/수정
  - 과거 출고/판매 실적 조회 및 비교
  - 단순 예측(최근 N개월 평균, 기초 추세) 제공(선택)
- **생산계획 관리**
  - 수요예측 데이터를 가져와 생산계획으로 전환
  - 제품/기간별 생산계획 수동 등록 및 수정
  - 계획 버전 관리(초안/확정)
- **BOM/소요량 계산**
  - 생산계획을 기준으로 자재 소요량 계산
  - 다단계 BOM 전개(필요 시) 및 기간별/품목별 소요량 집계
- **MRP/발주 계획**
  - 현재 재고, 안전재고, 리드타임을 고려한 자재 부족량 계산
  - 부족량 및 목표 납기일 기준 발주 제안(발주일, 발주수량)
- **발주 관리**
  - 발주 계획 리스트(제안 결과) 확인
  - 공급사별로 발주 계획을 묶어 발주서를 생성
  - 발주 상태 관리(요청/발주/입고완료)
- **재고/입출고 관리**
  - 품목/창고별 재고 현황 조회(안전재고 대비, 품절/과잉 표시)
  - 입고 처리(발주서 기준), 출고 처리(생산/판매/기타 사유)
  - 입출고 이력 조회 및 필터링
- **대시보드/리포트**
  - 품절 임박 품목, 지연 발주, 재고 현황 요약
  - 수요예측 vs 실제, 생산계획 vs 실제, 재고 추이 리포트

### 2.2 비기능 요구사항
- **성능**
  - 주요 목록 조회(재고, 발주, 생산계획, 수요예측)는 **3초 이내** 응답
  - MRP/BOM 계산은 중간 규모(수천 품목) 기준 **실용적인 시간 내(수십 초 이내)** 처리
- **보안**
  - 로그인 및 역할 기반 권한 관리(관리자/생산/구매/재고/조회 전용)
  - 주요 변경 이력(발주, 재고, 계획 등)은 추적 가능하도록 로그/히스토리 보관
- **확장성**
  - 향후 다른 시스템(ERP, WMS 등)과의 연동을 고려해 API 중심 설계
  - 테이블/엔티티 설계를 일반적인 SCM 구조에 맞게 확장 가능하게 정의
- **호환성**
  - 크롬/엣지 최신 버전 기준 웹 브라우저 지원
  - 엑셀 업로드/다운로드를 최소 1~2개 핵심 화면(품목, 생산계획 등)에 제공(선택)

## 3. 사용자 시나리오
### 시나리오 0: 신규 사용자의 회원가입 및 로그인
1. 사용자가 로그인 페이지에서 "회원가입" 버튼을 클릭한다.
2. 이메일/이름/비밀번호를 입력하고 가입을 완료한다.
3. 가입 직후 자동 로그인되거나, 로그인 페이지로 이동하여 이메일/비밀번호로 로그인한다.
4. 로그인 성공 후 대시보드 화면으로 이동하고, 본인 역할에 맞는 메뉴만 보인다.

### 시나리오 1: 생산 계획 담당자의 월간 계획 수립
1. 지난 6개월 판매/출고 실적을 조회한다.
2. 다음 달 제품별 수요예측 수량을 등록하거나, 간단 예측 기능을 사용해 자동 산출한다.
3. 수요예측 데이터를 기반으로 제품별 생산계획을 생성한다.
4. 필요 시 생산능력/제약을 고려하여 계획 수량을 조정하고, 계획을 ‘확정’ 상태로 변경한다.
5. 확정된 생산계획을 기준으로 BOM/소요량 계산 화면에서 자재 소요량을 확인한다.
6. 소요량이 확정되면 MRP를 실행해 자재 발주 제안을 생성한다.
7. MRP 결과를 구매 담당자에게 전달하거나, 시스템 내에서 바로 발주 계획으로 전송한다.

### 시나리오 2: 구매 담당자의 자재 발주 및 입고 처리
1. MRP 결과 화면에서 품목별 부족수량, 제안 발주일, 필요 납기일을 확인한다.
2. 우선순위(납기 임박, 품목 중요도 등)에 따라 발주 대상 품목을 선택한다.
3. 공급사별로 발주 계획을 묶어 발주서를 생성한다.
4. 공급사와 협의 후 납기일/수량을 필요 시 조정하고 발주 상태를 ‘발주’로 변경한다.
5. 실제 자재 입고 시, 발주서를 기준으로 입고 처리하고 재고가 자동으로 반영되는지 확인한다.
6. 지연 입고/미입고 건은 발주 현황 리포트에서 추적한다.

### 시나리오 3: 재고 담당자의 재고 모니터링 및 이슈 대응
1. 재고 현황 화면에서 품목/창고별 재고와 안전재고 대비 상태를 확인한다.
2. 품절/품절 예정/과잉재고 품목을 필터링해 리스트를 추출한다.
3. 품절 예정 품목은 생산/구매 담당자와 공유하여 추가 발주 또는 생산 조정 여부를 협의한다.
4. 과잉재고 품목은 프로모션 또는 생산계획 조정 등으로 소진 방안을 검토한다.

## 4. UI/UX 설계
### 4.1 화면 구성
- **인증 > 로그인/회원가입 화면**: 이메일/비밀번호 입력, 회원가입/로그인/비밀번호 재설정 진입점
- **대시보드 화면**: 재고 현황, 품절 임박 품목, 발주 지연, 수요/생산/재고 요약 카드 및 그래프
- **기초 데이터 > 품목 관리 화면**: 품목 리스트, 필터(코드/명/유형), 상세/등록/수정 팝업
- **기초 데이터 > BOM 관리 화면**: 제품 선택 후 하위 자재 리스트, 소요수량 관리
- **수요/생산 > 수요예측 화면**: 제품/기간별 예측 수량 입력 그리드, 과거 실적 조회
- **수요/생산 > 생산계획 화면**: 예측 불러오기, 계획 수량/상태(임시/확정) 관리
- **자재/MRP > 소요량 계산 화면**: 생산계획 선택 후 자재별 소요량 집계 결과
- **자재/MRP > MRP 결과 화면**: 품목별 필요량, 재고, 부족량, 제안 발주일/납기일
- **발주/구매 > 발주 계획/발주서 화면**: 제안 리스트 → 발주서 생성/상태 관리
- **재고 > 재고 현황/입출고 화면**: 재고 현황 리스트, 입고/출고 처리 폼, 이력 조회

### 4.2 주요 UI 컴포넌트
- **공통 검색/필터 바**: 품목코드/명, 기간, 상태(확정/임시/완료 등)를 빠르게 필터링
- **그리드/테이블 컴포넌트**: 컬럼 정렬, 필터, 페이징, 합계 표시 지원
- **상세/팝업 모달**: 품목, BOM, 발주서, 생산계획 상세 정보 확인 및 수정
- **알림/배지 컴포넌트**: 품절 임박, 지연 발주, 과잉재고 등 중요 이슈를 시각적으로 표시

### 4.3 사용자 플로우
```
[로그인]
  ↓
[대시보드] ————> [재고 현황 상세]
  ↓
[수요예측 관리] → [생산계획 수립] → [BOM/소요량 계산] → [MRP 결과]
                                                       ↓
                                                [발주 계획/발주서]
                                                       ↓
                                                [입고 처리/재고 반영]
                                                       ↓
                                                [출고/납품 처리]
```

## 5. 데이터 모델
### 5.1 주요 엔티티
- **품목(Item)**: 제품/원자재 마스터 정보
- **재고(Inventory)**: 창고/품목별 재고 수량
- **BOM(Bill of Materials)**: 상위 제품과 하위 자재의 구성/소요수량
- **리드타임(LeadTime)**: 품목·공급사별 조달 리드타임 및 최소 발주 수량
- **수요예측(DemandForecast)**: 제품/기간별 예측 수량
- **생산계획(ProductionPlan)**: 제품/기간별 생산 계획 수량 및 상태
- **MRP 결과(MaterialRequirement)**: 소요량·재고·부족량·제안 발주일/납기일
- **발주(PurchaseOrder)**: 발주서 헤더/라인 정보(공급사, 품목, 수량, 납기 등)
- **입출고 이력(StockTransaction)**: 입고/출고/조정 등 재고 변동 이력

### 5.2 데이터 관계
- 품목 1:N 재고
- 품목 1:N BOM(상위:하위)
- 품목 1:N 리드타임(공급사별)
- 제품 1:N 수요예측, 1:N 생산계획
- 생산계획 1:N MRP 결과
- 발주 헤더 1:N 발주 라인
- 품목 1:N 입출고 이력

## 6. 기술 스택
### 6.1 프론트엔드
- React 또는 Vue + TypeScript
- UI 라이브러리: Ant Design / MUI / Element 등 중 택1

### 6.2 백엔드
- Python FastAPI + Uvicorn
- 도메인 단위 Router(품목, 재고, 수요예측, 생산계획, MRP, 발주 등)로 분리
- Render/Railway/Fly.io 중 하나에 컨테이너 또는 Web Service 형태로 배포
 - 인증: JWT 발급/검증, 비밀번호 해시(bcrypt) 처리

### 6.3 데이터베이스
- PostgreSQL (Supabase/Render DB/Neon 등 매니지드 서비스)
- ERD 기준으로 테이블 설계 후 마이그레이션 관리(예: Alembic)

### 6.4 외부 서비스
- (필수) GitHub + Vercel: `frontend` 자동 빌드/배포
- (추천) Render: `backend` FastAPI 자동 배포
- (필수) Supabase/Neon: PostgreSQL 관리
- (선택) Cloudflare R2/AWS S3: Excel·첨부 파일 저장

## 7. 수용 기준 (Acceptance Criteria)
- [ ] 수요예측 → 생산계획 → MRP → 발주 → 입고 → 재고 반영의 기본 플로우를 한 화면 흐름에서 수행할 수 있다.
- [ ] 품목/재고/BOM/발주 등 핵심 데이터가 UI/리포트에서 일관되게 보인다.
- [ ] 주요 리스트 화면(재고, 발주, 생산계획, 수요예측)의 응답 시간이 3초 이내이다.
- [ ] 역할별 권한(관리자/생산/구매/재고/조회 전용)이 정의·적용되어 있다.

## 8. 가정 (Assumptions)
- 회사 내 기본 마스터 데이터(품목/거래처/창고)는 엑셀 또는 기존 시스템을 통해 확보 가능하다.
- 초기에는 단일 공장/창고 기준으로 운영하며, 다공장/다창고는 단계적으로 확장한다.
- 예측 정확도보다는 **업무 프로세스 정착 및 가시화**를 1차 목표로 한다.

## 9. 제약사항
- 전사 ERP/회계 시스템과의 실시간 연동은 초기 범위에 포함하지 않는다.
- 인프라/서버 자원은 제한적이므로, 과도한 실시간 대용량 처리보다는 배치/오프라인 계산을 우선 고려한다.
- 사용자 수는 소수(수~수십 명)로 가정하고 설계한다.

## 10. 리스크
- **요구사항 변경 리스크**: 업무 부서별로 요구사항이 빈번하게 변경될 수 있음 → 초기에는 ‘경량 버전’에 집중하고, 확장 기능은 별도 로드맵으로 분리.
- **데이터 품질 리스크**: 기존 재고/품목 데이터의 정확도가 낮을 수 있음 → 초기 데이터 정비 및 검증 작업을 프로젝트 범위에 포함.
- **도입 저항 리스크**: 엑셀 중심 업무 문화로 인해 시스템 도입에 저항이 있을 수 있음 → 주요 사용자에 대한 교육 및 단계적 적용 전략 수립.
